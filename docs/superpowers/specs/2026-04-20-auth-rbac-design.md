# Design: Auth + RBAC MVP (Approach 2)

**Date:** 2026-04-20
**Status:** Approved for implementation planning
**Scope:** Magic-link authentication, 3-role RBAC (admin / broker / client), client-org tenancy, URL-scoped routes, simple credit counter, admin provisioning UI.
**Related backlog items:**
- Implements: BL-009 (partial ‚Äî magic link instead of Entra), BL-010, BL-040 (partial ‚Äî counter only), BL-043, BL-044 (partial ‚Äî fake vouchers), BL-045
- Deferred to: BL-041 (real vouchers), BL-047 (Entra SSO), BL-048 (credit audit ledger), BL-049 (landing-page port)

---

## 1. Goals

Ship auth + RBAC + tenancy that is good enough for internal testing across three distinct role behaviors (Don + cover.mesh + 2‚Äì3 SCHUNCK colleagues), using magic-link email login and admin-provisioned accounts. Data is isolated per client org so role-gated UX can be validated end-to-end. The simple integer credit counter makes the BL-043/044 balance displays real without requiring a full audit ledger yet.

## 2. Non-goals

- OAuth / SSO of any kind (Entra, Google, Microsoft) ‚Äî tracked in BL-047.
- Self-signup, password login, MFA, account recovery.
- Per-debit credit audit ledger ‚Äî tracked in BL-048.
- Real voucher generate/redeem/expire flow ‚Äî tracked in BL-041; MVP ships fake UI only.
- Landing-page port ‚Äî tracked in BL-049; MVP serves a minimal placeholder at `/`.
- Broker-to-client portfolio assignments (Q6 answer: flat access for MVP).
- Preserving the ~27 existing checks in prod Turso (wiped during migration).

## 3. Stack

- **Auth library:** Auth.js v5 (`@auth/core`, `@auth/drizzle-adapter`).
- **Email delivery:** Resend (via Auth.js `Email` provider). `RESEND_API_KEY` + `EMAIL_FROM` env vars added to local `.env.local` and Vercel Production/Preview.
- **Session storage:** Database-backed via Drizzle adapter (Auth.js `DatabaseSession` strategy). Session cookie: HTTP-only, `Secure`, `SameSite=Lax`, 30-day rolling expiry.
- **Middleware:** `src/middleware.ts` (Next.js middleware running on every matched route) reads session, resolves role + client scope, enforces gating.
- **DB:** Existing Turso libSQL via `@libsql/client` + `drizzle-orm/libsql`. No storage changes.

## 4. Data model

### New tables

**`users`**
| Column | Type | Notes |
|---|---|---|
| `id` | text PK | Auth.js-generated |
| `email` | text unique not null | Login identifier |
| `name` | text | Display name, optional |
| `role` | text enum `("admin","broker","client")` not null | Role gating key |
| `client_id` | text FK ‚Üí `clients.id` nullable | Required iff `role = "client"`; null for admin/broker |
| `created_at` | text not null | ISO timestamp |

**`clients`**
| Column | Type | Notes |
|---|---|---|
| `id` | text PK | App-generated |
| `slug` | text unique not null | URL-safe (`/client/[slug]/...`) |
| `name` | text not null | Display name (firm name) |
| `credit_balance` | integer not null default 0 | Simple counter, decremented per scan |
| `created_at` | text not null | |

**Auth.js adapter tables** (framework-managed, schemas generated by `@auth/drizzle-adapter`):
- `accounts` ‚Äî OAuth links (empty for magic-link-only MVP but schema must exist).
- `sessions` ‚Äî session id + userId + expires.
- `verification_tokens` ‚Äî magic-link tokens (identifier + token + expires).

### Modified tables

**`checks`** ‚Äî two new NOT NULL columns:
| Column | Type | Notes |
|---|---|---|
| `client_id` | text FK ‚Üí `clients.id` not null | Scope for data isolation |
| `created_by_user_id` | text FK ‚Üí `users.id` not null | Actor attribution (for broker activity) |

### Migration

Single Drizzle migration:
1. Create `users`, `clients`, `accounts`, `sessions`, `verification_tokens`.
2. Wipe existing scan data: `DELETE FROM feedback; DELETE FROM chat_messages; DELETE FROM documents; DELETE FROM checks;` (order respects FKs).
3. Alter `checks` to add `client_id` + `created_by_user_id` NOT NULL (table is now empty, so no backfill needed).
4. Clear Blob store: delete all keys under `checks/` prefix.

The existing local SQLite fallback DB stays compatible ‚Äî same migration applies.

### Bootstrap

`scripts/seed-admin.ts`:
- Reads `SEED_ADMIN_EMAIL` + `SEED_ADMIN_NAME` from env (default `don.rorlach@covermesh.com`).
- Inserts one `users` row with `role="admin"` if not present.
- Idempotent; safe to re-run.

Runs once after migration. CLI only ‚Äî no API endpoint for bootstrap (chicken-and-egg: a gated admin route can't be used when no admin exists yet). On prod: `vercel env pull` to get `TURSO_DATABASE_URL` + `TURSO_AUTH_TOKEN`, then `tsx scripts/seed-admin.ts`. Script logs the inserted user's email.

| `/api/analytics` | Admin only | If any external consumer exists, this flag can be revisited |

### Middleware logic

```
1. If request.path starts with /api/auth, /login, /, or /_next ‚Üí pass through.
2. Get session from cookie.
3. If no session and path is protected ‚Üí redirect to /login?next=<encoded-current-path>.
4. If session exists and path == /login or / ‚Üí redirect to role-start page (/admin | /broker | /client/<slug>).
5. Role gate: compare session.role against required role set for the matched route.
6. Slug gate (role=client only): extract `[slug]` from `/client/[slug]/...`, compare to session.user.clientId ‚Üí clients.slug. Reject 403 on mismatch.
```

Middleware runs on all non-static routes; API routes additionally re-verify inside the handler (defense in depth, because middleware can't parse request bodies).

## 6. UX flows

### Magic-link login

1. User loads `/login` ‚Üí enters email ‚Üí POST to `/api/auth/signin/email`.
2. Auth.js generates a 1-hour single-use token, stores it in `verification_tokens`, calls Resend with our `EMAIL_FROM`.
3. UI shows "Pr√ºfen Sie Ihr Postfach" regardless of whether the email matched a user.
4. User clicks the link ‚Üí Auth.js verifies the token, creates a `sessions` row, sets the cookie, redirects to `/`.
5. `/` middleware sees the session and redirects to `/admin`, `/broker`, or `/client/<slug>` based on role.

Unknown email path: Auth.js default behavior does not reveal non-existence; we additionally patch the flow to skip calling Resend when the email is not in `users` (avoids sending to arbitrary addresses).

### Admin creates a client org + first user

1. Admin navigates to `/admin/clients/new`.
2. Fills: firm name, slug (auto-derived from name, editable), initial credit balance (integer), first-user email + name.
3. Submit ‚Üí server inserts `clients` row + `users` row (role=client, client_id set).
4. Admin is shown the first-user email. No automatic email is sent in MVP; admin manually informs the user that they can log in via `/login` with that email. (Rationale: avoids configuring transactional-invite templates; internal testing only.)

### Admin creates an admin or broker user

1. `/admin/users/new` ‚Üí email, name, role (admin | broker) ‚Üí submit ‚Üí `users` row inserted with `client_id = null`.

### Broker scan

1. Login ‚Üí redirect to `/broker`.
2. Start page (BL-044) shows: searchable list of all `clients`, a "Voucher generieren" tile (fake), and an instruction paragraph.
3. User searches + picks "Acme GmbH" ‚Üí router pushes `/client/acme-gmbh/check`.
4. Upload / classify runs as today, scoped to that client.
5. Clicking "Vollst√§ndige Analyse starten" hits `POST /api/analyze`; server checks `clients.credit_balance >= tierCost`:
   - If insufficient ‚Üí responds `402 Payment Required` with error; UI shows "Kein ausreichendes Guthaben (X Credits ben√∂tigt, Y verf√ºgbar)".
   - If sufficient ‚Üí runs analysis; on success, atomic `UPDATE clients SET credit_balance = credit_balance - <cost> WHERE id = <clientId> AND credit_balance >= <cost>` ensures no negative balance under concurrent requests.
6. Results at `/client/acme-gmbh/results/[id]`.

Note: per Q6 answer, MVP treats all brokers as equal ‚Äî any broker can act on any client.

### Client scan

1. Login ‚Üí redirect to `/client/<own-slug>`.
2. Start page (BL-043) shows: credit balance prominently, one-paragraph instructions, primary "Neue Pr√ºfung" CTA.
3. Click ‚Üí `/client/<own-slug>/check` (no picker, scope is implicit).
4. Same server-side credit debit as broker flow.

### Admin scan

1. Login ‚Üí redirect to `/admin`.
2. Start page (BL-045) shows: stats block (users count, clients count, total credits in circulation, scans this month), shortcuts to `/admin/clients`, `/admin/users`, and a client-picker tile identical to broker's (admin can also run scans for any client).

## 7. Credit mechanics (counter-only)

Tier costs:

```ts
// src/lib/credits.ts
export const TIER_COSTS = { quick: 1, medium: 2, full: 5 } as const;
```

Debit is inside `/api/analyze` route handler, wrapped in a single SQL UPDATE that includes the `credit_balance >= cost` guard clause. If the UPDATE affects 0 rows, we return 402 without running analysis. This makes concurrent-scan races impossible.

Admin setting the balance: `POST /api/admin/clients/<id>` with `{creditBalance: <n>}` is an idempotent set (not a delta). No audit row recorded ‚Äî that's BL-048's job.

Fake voucher UI: a `<form>` on `/broker` with an input labeled "Menge" and a "Code generieren" button. On click, client-side JS generates a random 8-character alphanumeric string and displays it in a readonly field with a copy button. Zero server interaction.

## 8. Security posture

- Sessions are HTTP-only cookies; not readable from JS. Secure flag on in production.
- Magic-link tokens are single-use (Auth.js default) and expire after 1 hour.
- Rate limit `/api/auth/signin/email` via Vercel's built-in rate limiter or a lightweight in-DB counter (5 requests per email per hour) ‚Äî prevents someone flooding a real user's inbox.
- Role and slug checks are enforced at **both** middleware and API-handler levels (defense in depth).
- Admin API routes double-check `session.user.role === "admin"` at the top of every handler.
- No logging of magic-link tokens, session IDs, or full email bodies.
- Error messages in server logs do not include user email in cases where the email was not found (avoids an enumeration side channel).

## 12. Open questions / risks

- **Resend domain verification delay** ‚Äî first-time DNS TXT + DKIM can take 30 min. Set up in parallel with implementation, not on the critical path.
- **Auth.js v5 + Next.js 16 App Router** ‚Äî v5 is GA but App Router integration still has sharp edges on Next 16 specifically. Budget an hour for first-bump debugging.
- **Middleware vs API-handler duplication** ‚Äî every check is enforced in both layers; this is intentional (middleware can't parse request bodies reliably). Expressed as a single `requireRole(session, ...)` helper used in both.
- **"Admin tells user to log in manually"** is a UX papercut for MVP. Tracked as a minor follow-up: proper invite emails once Resend is configured and we have a template.
